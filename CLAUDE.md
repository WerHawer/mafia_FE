# FE Project Rules (Mafia)

Скопійовано з `.cursor/rules/` — застосовується завжди.

## Core Standards (`fe-core-standards`)

### Роль і комунікація

- Працюй як senior front-end: React, TypeScript, SCSS modules, MobX, Vite, react-router.
- На відповідях користувачу дотримуйся правил чату (наприклад, префікс 🤖 там, де це зафіксовано в проєкті).

### Загальне

- Чітко з'ясуй вимоги; для складних задач спершу план (псевдокод / кроки), потім код.
- DRY, читабельність важливі за скорості; функціональні компоненти та хуки, без class components.
- Повна реалізація: без TODO, плейсхолдерів і недороблених шматків.
- Якщо відповіді немає або вона невпевнена — скажи прямо, не вигадуй.

### Tech stack і середовище

- React, JS/TS, SCSS modules, HTML, **Yarn** (не npm).
- Дев-сервер зазвичай уже запущено: `http://localhost:5173/` — не наполягай самостійно запускати застосунок, якщо користувач цього не просить.

### CRITICAL: verify before use

Перед використанням файлу, методу store, хука, бібліотеки:

1. Прочитай актуальний вміст файлу / експорти.
2. Перевір сигнатури, типи, форми даних.
3. Подивись пов'язані файли й приклади використання в репозиторії.
4. Не покладайся на здогадки з імен.

Пам'ятай: прочитати файл — секунди; виправляти хибні припущення — години.

### CRITICAL: verify after completion

Після завершення задачі (перед звітом користувачу):

1. **Перевір імпорти**: переконайся, що всі використані функції/хуки (наприклад, `useMemo`, `useEffect`, `useCallback`) імпортовані з відповідних модулів.
2. **Перевір змінні**: переконайся, що всі використані змінні оголошені та доступні в області видимості.
3. **Чистота коду**: видали невикористані імпорти або змінні, що залишилися після редагування.
4. **Типізація**: переконайся, що типи аргументів та повернень відповідають очікуванням.

### Якість коду

- Перед реалізацією: сумісність залежностей, патерни проєкту, витоки / ефективність.
- Самодокументні імена; коментарі — коли є «чому», складна логіка, магічні числа (краще іменовані константи).
- Строгий TypeScript: типи для параметрів і повернень; edge cases і валідація де доречно.

### Cross-file

- Типи з узгоджених файлів типів; **без barrel `index.ts`** — прямі імпорти з конкретних файлів.
- Абсолютні імпорти з кореня проєкту (`@/...`).
- Уникай циклічних залежностей; дотримуйся структури папок репозиторію.

### Продуктивність

- `useMemo` / `useCallback` / `React.memo` та cleanup в `useEffect` там, де є сенс.
- Лейзі-роути/компоненти, коректні `key` у списках; віртуалізація для довгих списків.

## React, TS і UI (`fe-react-ui`)

### Стиль коду

- Ранні `return`; **порожній рядок перед `return`** у функціях.
- `const` і стрілкові функції; явні типи для публічних функцій і пропсів.
- Обробники подій: префікс **`on`** (`onClick`, `onSubmit`), не `handle*` (виняток — проп з іншою назвою).

### Імпорти та store

- Не `React.useState` — імпортуй `import { useState, ... } from "react"`.
- Кілька звернень до одного store / `rootStore` — деструктуризація (`const { myId } = usersStore`). Детальніше: правило нижче (`destructuring`).

### Компоненти

- Перед новим UI: глянь `src/UI/`, потім `src/components/` — перевикористовуй існуюче.
- Композиція, один фокус відповідальності; коректний cleanup підписок.
- Де доречно: Suspense, error boundaries, зрозумілі стани завантаження / помилки.

### Доступність

- Інтерактив: клавіатура (`onKeyDown` де треба), `tabIndex={0}` коли логічно, `aria-label` для не-тексту, осмислені ARIA-ролі.
- Курсори: pointer для клікабельного, default для пасивного.

### SCSS modules

- `ComponentName.module.scss` поруч із компонентом.
- Тема й міксини з `src/styles`.
- Імена класів відображають призначення; дотримуйся наявних патернів.

### classnames

```tsx
import classNames from "classnames";

className={classNames(styles.container, styles[variant], { [styles.active]: isActive })}
```

Уникай сирих шаблонних рядків для складних комбінацій класів.

### Інтерактив

- Для складних контролів: `useCallback`, мемоізація за потреби; refs для dropdown/tooltip; клік поза елементом де прийнято в проєкті.

## Деструктуризація (`destructuring`)

Коли з одного об'єкта зчитуєш кілька полів (`rootStore`, `gamesStore`, контекст, великий props), **деструктуризуй на початку** функції / хука / компонента замість розкиданих звернень `obj.field` у новій або зміненій логіці.

```typescript
// ❌ Гірше: розкидані звернення
const { gamesStore } = rootStore;
if (gamesStore.isMeObserver && gamesStore.gameFlow.isNight) { ... }

// ✅ Краще
const { gamesStore, isIGM } = rootStore;
const { isMeObserver, gameFlow } = gamesStore;
if (isMeObserver && gameFlow.isNight) { ... }
```

- Нові залежності від store/context додавай через деструктуризацію в тому ж блоці, що й інші поля цього джерела.
- Не змішуй без потреби: `const x = store.a; const y = store.b;` якщо достатньо `const { a: x, b: y } = store`.
- Типовий виняток — **вибірковий доступ** у рідких гілках або коли імена конфліктують (тоді перейменування при деструктуризації).

## API і дані: TanStack Query (`fe-tanstack-query`)

Усі запити до API оформлюй через **TanStack Query (React Query)**.

### Обов'язкові практики

- Читання: **`useQuery`** з передбачуваними **query keys**.
- Зміни: **`useMutation`** з коректною **інвалідацією** кешу після успіху.
- Обробка помилок і **retry** за патерном проєкту; за потреби **prefetch**.
- Типи відповідей і змінних запиту — у TypeScript.

### Структура коду

- Окремі файли для хуків запитів / мутацій (наприклад `useXQuery.ts`, `useXMutation.ts`).
- Групуй по фічах у каталогах; узгоджена обробка помилок.

### Поведінка кешу

- Розумні `staleTime` / `gcTime` (раніше cacheTime) узгоджуй з існуючими query.
- Після мутацій: оновлення кешу або invalidation; **optimistic updates** там, де це вже прийнято.
- За потреби: `placeholderData` / `initialData`, `select` для трансформації на клієнті.

### Складні сценарії

- Залежні й паралельні query; **infinite query** для пагінації, якщо фічі це вимагає.
- Polling і Suspense — лише коли відповідає архітектурі екрану.

MobX store для клієнтського стану гри не замінюй Query-шаром без потреби — дотримуйся поточного поділу в репозиторії.

## Git: коміти (`fe-git-commits`)

Усі **`git commit` повідомлення** — **англійською**: і **subject** (перший рядок), і **body** (опис із нових рядків при потребі).

- Не використовувати українську, російську тощо в тексті коміту.
- Префікс типу коміту за узгодженим у команді патерном (наприклад **Conventional Commits**: `feat:`, `fix:`, …) залишається англійською й узгоджується з існуючою історією репозиторію.

Пояснення для користувача в чаті можуть бути українською за правилами чату проєкту; обмеження стосується саме тексту коміту в Git.
